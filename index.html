Can Machines Detect Sarcasm
Across Different Domains?

Sarcasm is one of the hardest linguistic phenomena for machines to detect — it requires understanding intent, tone, and context, not just the words themselves. In this project we investigate whether classical machine learning methods, when combined with carefully engineered features, can reliably identify sarcasm across two very different domains: formal news headlines and informal online forum comments. Along the way we uncover a critical data quality problem — source leakage — and show how training on a single-source dataset can fool a model into learning the wrong thing.

01 — Introduction

Why Is Sarcasm Hard for Machines?

When a person says "Oh great, another Monday", the words themselves are positive — but the meaning is negative. Humans understand this instantly through tone, context, and cultural knowledge. Machines, however, only see the text. This makes sarcasm detection a fundamentally challenging natural language processing problem.

Standard sentiment analysis tools like VADER and TextBlob are designed to score the emotional content of words. A sentence packed with positive vocabulary ("great", "amazing", "wonderful") will receive a high positive score — even if the author intended the opposite. This project investigates just how often this happens, and whether machine learning models trained on word patterns and stylistic signals can do better.

We work with two datasets from different domains: formal news headlines (The Onion vs HuffPost) and informal Usenet forum comments. This cross-domain design allows us to test whether models learn genuine sarcasm signals or simply memorise the writing style of a particular source — a distinction that turns out to be critical.

What We Did — Project Overview

  1. Loaded and explored the raw data
    Checked for duplicates, missing values, class balance, and basic statistics on both the news headlines and forum comments datasets.
  2. Preprocessed both datasets
    Extracted six structural features from the raw text (character count, word count, average word length, quote count, exclamation count, question mark count), then cleaned the text (lowercasing, removing punctuation and digits, filtering short texts).
  3. Exploratory data analysis
    Visualised class distributions, feature distributions by class, correlations, and top word pair patterns in sarcastic vs normal texts.
  4. Sentiment analysis (RQ2)
    Applied VADER and TextBlob to both datasets to measure how often standard sentiment tools misclassify sarcastic content as positive.
  5. News-only baseline models (RQ1, RQ3)
    Trained Logistic Regression, Linear SVM, and Random Forest on the news headlines dataset across five different feature sets. Used 5-Fold Cross-Validation to evaluate each combination.
  6. Leakage detection and cross-domain test
    Discovered that the best news model had learned the writing style of The Onion rather than genuine sarcasm signals. Confirmed this by testing the model on forum comments — performance collapsed.
  7. Domain-independent combined model
    Combined both datasets, retrained all models using Nested Cross-Validation, and evaluated on a held-out test set to produce honest, leakage-free results.
02 — Dataset

Two Datasets, Two Very Different Domains

We use two publicly available datasets. Using two domains is deliberate — it allows us to test whether models learn genuine sarcasm patterns or domain-specific writing styles.

News Headlines Dataset

The News Headlines Dataset for Sarcasm Detection contains headlines from two sources: The Onion (a satirical publication — all sarcastic) and HuffPost (a factual news site — all non-sarcastic). Headlines are short, averaging around 10 words, with a formal journalistic style.

Forum Comments Dataset (GEN)

The Sarcasm Corpus V2 (Oraby et al., 2016) contains comments from Usenet discussion forums. These are much longer (~35 words on average), informal, and written in a conversational style — a very different domain from news headlines. We use the GEN-sarc-notsarc subset, which is balanced 50/50.

03 — Preprocessing

Preparing the Data for Machine Learning

Both datasets go through the same preprocessing pipeline, ensuring fair comparison. The most important decision here is the order of operations: structural features must be extracted before text cleaning, because cleaning removes punctuation — and punctuation is itself a feature.

  1. Duplicate removal and column cleanup
    Exact duplicate texts are dropped. Irrelevant columns (article links, IDs) are removed. Labels are converted to 0 (normal) and 1 (sarcastic).
  2. Structural feature engineering (from raw text)
    Six numerical features are extracted before any cleaning: character count, word count, average word length, quote count, exclamation mark count, and question mark count. These capture how something is written, not what it says.
  3. Text cleaning
    Contraction expansion (e.g. "don't" → "do not") → lowercase conversion → digit removal → punctuation removal → whitespace normalisation.
  4. Integrity filtering
    Texts shorter than 3 words after cleaning are removed. This removed 219 rows from the news dataset and 0 from the forum dataset.
  5. Dataset combination
    Both cleaned datasets are merged into a single combined dataset. A source column is added to track which domain each text came from. The combined dataset is shuffled with a fixed random seed (42) for reproducibility.

04 — Methodology

How We Approached the Problem

We frame sarcasm detection as a binary classification task: given a text, predict whether it is sarcastic (1) or normal (0). We deliberately chose classical machine learning methods — not deep learning — to understand which features and model types carry the most signal, and to keep results interpretable and reproducible.

Feature Sets

We experiment with five different ways of representing each text as numbers:

Stylistic Features

The 6 structural features extracted during preprocessing: character count, word count, average word length, quote count, exclamation count, question mark count. These capture writing style — how something is written.

Word Frequency (Single Words)

Each unique word in the vocabulary becomes a column. The value is the word's frequency in that text, adjusted for how rare the word is across all texts (this is called TF-IDF weighting). Captures what is said.

Word Frequency + Stylistic Features

Combines single-word frequency with the 6 stylistic features. This is our richest feature set and the best performer overall.

Word Pairs Only

Instead of single words, we count pairs of consecutive words (e.g. "area man", "study finds"). Word pairs capture some local context that single words miss.

Words & Pairs

Combines single words and word pairs in the same feature matrix. Tests whether adding word pairs on top of single words provides meaningful additional signal.

Machine Learning Models

We train three different models on each feature set, chosen to represent different approaches to classification:

Logistic Regression (LR)

A linear model that assigns a weight to each feature and predicts the probability of sarcasm. Fast, interpretable (we can read which words matter most), and well-suited to high-dimensional word frequency data. We use this for leakage analysis because its weights are directly readable.

Linear SVM

Finds the widest possible boundary between sarcastic and normal texts in feature space. Linear SVMs are particularly effective for high-dimensional sparse data like word frequencies, often matching or slightly improving on Logistic Regression.

Random Forest (RF)

An ensemble of decision trees, each trained on a random subset of features. Unlike the linear models above, Random Forest can capture non-linear relationships. We include it especially to test stylistic features, where its tree-based structure handles numerical data naturally.

Evaluation Strategy

Why not just split data into train and test?
A single split can be unlucky — the test set might be easier or harder than average. We use Cross-Validation instead: the training data is split into 5 equal parts, and the model is trained and tested 5 times, each time using a different part as the test fold. The average F1 score across all 5 folds is much more reliable.
What is Nested Cross-Validation (used in the combined model)?
When we also need to tune model parameters (e.g. how strong the regularisation should be), a simple cross-validation loop would leak information — we'd be using the test fold to choose parameters. Nested CV solves this: an inner loop finds the best parameters using only the training portion, and an outer loop evaluates the result on held-out data. This gives an unbiased performance estimate.
Why do we split 80/20 if we already use Cross-Validation?
Cross-Validation runs entirely within the training set — it tells us which model and parameters work best on data the model has already been exposed to. The moment we use cross-validation results to choose a model, that choice is influenced by the training data. The held-out 20% test set answers a different question: "How does the chosen model perform on data it has never seen at all?" This is the only truly unbiased performance estimate. The two steps work together: Nested CV selects the best model; the test set proves it.

05 — Step 1: Baseline on News Headlines

Testing 5 Feature Sets × 3 Models on News Headlines

We start by building baseline models on news headlines only. This answers our three research questions and establishes a performance ceiling — before we discover the leakage problem in Step 2.

Research Question 1
1
Does sarcasm reveal itself more through what is said (word content) or how it is said (punctuation, sentence length, writing style)?

Methodology

We train all three models (Logistic Regression, Linear SVM, Random Forest) on five feature sets on the news headlines dataset using 5-Fold Cross-Validation. This tells us which feature type carries a stronger sarcasm signal within a single domain. We then test the same models on forum comments (an unseen domain) to see which features generalise better.

Interpretation

Within news headlines: Word frequency signals (F1: 0.819) far outperform stylistic features (F1: 0.497). The specific words used in a headline carry a strong sarcasm signal within this domain.
Across domains: The picture reverses completely. On forum comments, stylistic features (F1: 0.363) dramatically outperform word frequency features (F1: 0.120). Stylistic signals generalise; word-content signals do not — because words that signal sarcasm in news headlines may be completely different from those used in forum comments.
Answer to RQ1: Neither approach alone is sufficient. Word content wins in a single domain; writing style wins across domains. The best model combines both features and achieves a test F1 of 0.787 on the combined dataset — confirming that sarcasm involves both what is said and how it is said. Notably, Random Forest outperforms linear models on stylistic features alone (0.506 vs 0.497), because tree-based models handle numerical features more naturally.
Research Question 2
2
How and how often do standard sentiment analysis tools mistakenly score sarcastic content as positive — and does this problem get worse in longer, more conversational texts?

Methodology

We apply two widely-used sentiment analysis tools — VADER and TextBlob — to both datasets separately. Both tools work by assigning positive or negative scores to individual words and aggregating them to produce an overall sentiment score. We measure the "misleading rate": the percentage of genuinely sarcastic texts that each tool incorrectly scores as positive.

We also examine the distribution of sentiment scores and measure how often inherently positive words ("great", "amazing", "wonderful") appear in sarcastic vs normal texts.

Most Deceiving Headline

VADER score: 0.915 (near maximum positive). Words like "fair", "free", and "loyal" trigger a high positive score — yet the sentence is clearly sarcastic.

Why Forum Comments Are Harder for Sentiment Tools

The misleading rate is substantially higher for forum comments (VADER: 46.8%, TextBlob: 49.5%) than for news headlines (VADER: 31.4%, TextBlob: 31.6%). There are two reasons for this:

  • Length effect: Forum comments average ~35 words, news headlines ~10 words. VADER and TextBlob work by adding up the sentiment scores of individual words. A longer text has more chances to include positive-sounding words — even if the overall intent is sarcastic. More positive words → higher positive aggregate score → more likely to be misclassified.
  • Style effect: Forum sarcasm is often more subtle and elaborate. Sarcastic news headlines tend to be direct parodies ("Area Man Does X"). Forum comments often build a long argument before revealing the sarcasm, giving sentiment tools more misleading positive vocabulary to latch onto.

Interpretation

Answer to RQ2: Both tools fail at structurally similar rates — VADER and TextBlob perform nearly identically. This is not a tool-specific bug; it is a fundamental limitation of word-level sentiment scoring. Sarcasm intentionally uses positive vocabulary to convey negative meaning, and any tool that ignores sentence-level context will be systematically deceived. The problem is worse in longer texts because the accumulation of positive words overwhelms the negative intent.
Research Question 3
3
Does capturing word pairs (e.g. "area man", "study finds") meaningfully improve sarcasm detection beyond looking at single words alone?

Methodology

We compare three word-frequency configurations: single words only, word pairs only, and both combined. All three are tested with all three models on both datasets. Word pairs capture local context that single words miss — for example, "area man" is a much stronger sarcasm signal than "area" or "man" individually.

Why test word pairs? Sarcasm often relies on specific phrase patterns. The Onion famously uses templates like "Area Man Does X" or "Nation's [Group] Reports Y". If these patterns are informative, word pairs should outperform single words.

Interpretation

Word pairs alone underperform. At F1 = 0.662 (Logistic Regression, news), word pairs alone are substantially weaker than single words. Without single-word context, the model loses too much vocabulary coverage.
Combining words and pairs gives minimal improvement. Adding word pairs to single words improves F1 by only 0.002 on news headlines and 0.005 on the combined dataset — practically no difference.
Answer to RQ3: Word pairs alone are not sufficient, and adding them to single words provides only marginal improvement. Sarcasm detection requires understanding broader context — the intent of an entire sentence, cultural references, and tonal cues that go far beyond two-word combinations. This points to a fundamental ceiling of bag-of-words approaches, whether unigrams or bigrams.
06 — Step 2: Did the Model Learn the Right Thing?

Leakage Detection — The Model Learned Writing Style, Not Sarcasm

The baseline achieved F1 = 0.827 on news headlines. Before moving on, we must ask: what did the model actually learn? The answer reveals a critical problem — and motivates everything in Steps 3 and 4.

After achieving strong results on news headlines (F1: 0.827), we investigated what the model actually learned. The answer was unexpected — and revealed a fundamental problem with training on single-source data.

Section B — Why We Did Coefficient Analysis

Logistic Regression assigns a numerical weight (coefficient) to every word in its vocabulary. A high positive coefficient means the word strongly predicts sarcasm; a high negative coefficient means the word strongly predicts normal content. By reading the top coefficients, we can see exactly which words the model treats as sarcasm signals.

Fig 9 — Words with the highest weights in the Logistic Regression model | Word Frequency (Single Words) feature set

The word "onion" ranks 14th among the most predictive sarcasm words (weight: 3.09). Words like "area", "man", "nation", "report", and "local" — all characteristic of The Onion's distinctive writing style — dominate the top of the list. The model has learned to recognise The Onion's voice, not sarcasm itself.

Why Coefficient Analysis Alone Is Not Enough

Seeing "onion" in the top tokens is suspicious, but it could be coincidental. Perhaps "onion" genuinely co-occurs with sarcasm for linguistic reasons we haven't considered. The coefficient analysis gives us a strong hypothesis — but a hypothesis is not proof. We need a proper test.

Section C — The Proof: Testing on an Unseen Domain

If the model learned genuine sarcasm signals, it should work on any sarcastic text — not just The Onion's headlines. So we tested it on the forum comments dataset, which the model had never seen and which has no connection to The Onion.

Word frequency collapses completely: F1 drops from 0.819 to 0.120 — barely above random guessing. The model that seemed excellent on news headlines is essentially useless on forum comments. This is the proof: if it had learned sarcasm, it would still detect sarcasm in forum comments.
Stylistic features are the most resilient: F1 drops only from 0.497 to 0.363 — a much smaller fall. Punctuation patterns and sentence length are more universal sarcasm signals that transfer across domains.
Words & Pairs shows the largest drop (−0.708): The more vocabulary the model learns, the more domain-specific patterns it memorises, and the worse it performs on an unseen domain.
Conclusion: The coefficient analysis (Section B) and the cross-domain test (Section C) together confirm source leakage. The model learned The Onion's writing style, not sarcasm. Section D addresses this by training on data from both domains.
07 — Step 3: Fix Leakage with Combined Training Data

Training on News Headlines + Forum Comments Together

We know the news-only model learned The Onion's writing style. The fix: train on both domains simultaneously. A model that sees sarcasm from news headlines and forum comments can no longer rely on source-specific vocabulary — it must find signals that work in both.

To address the leakage problem, we combine both datasets — news headlines and forum comments — into a single training corpus. By seeing sarcasm from two very different domains, the model can no longer rely on source-specific vocabulary. It must find signals that work in both.

Setup

  • 33,010 texts combined (80% news, 20% forum)
  • Stratified 80/20 train/test split — the 20% test set is set aside and never touched until final evaluation
  • Numerical stylistic features are standardised (StandardScaler fitted on training data only, then applied to test — prevents data leakage)
  • Nested Cross-Validation on the training set: inner loop optimises model parameters, outer loop evaluates performance
  • All 5 feature sets × 3 models = 15 experiments

08 — Step 4: Is the Model Truly Domain-Independent?

Domain-Specific Testing and Balanced Training

The combined model achieves F1 = 0.787 overall — but is this score equally good on news headlines and forum comments? We split the held-out test set by source and evaluate separately. None of these rows were seen during training.

4b — Balanced Training Experiment

To test whether balancing training data closes the gap, we downsample news headlines to match the forum size (6,520 each) using stratified sampling — preserving the sarcastic/normal ratio. The train/test split is done after downsampling, so the test set was never seen during training.

Balanced model closes the domain gap almost completely: News F1 = 0.727 vs Forum F1 = 0.725 — a difference of only 0.002. The model now performs equally well on both domains, confirming that the gap in the imbalanced model was caused by training data imbalance, not an inherent difference in task difficulty.
Trade-off — lower overall F1: The balanced model achieves F1 = 0.726 vs 0.787 for the imbalanced model. Training on ~13,000 texts instead of ~33,000 means less data overall, which reduces overall performance. The choice depends on the goal: higher average performance (imbalanced) or fairer cross-domain performance (balanced).
Key insight: Domain balance in training data directly controls domain balance in model performance. For any real-world sarcasm detection system deployed across multiple domains, balanced multi-domain training is essential — even at the cost of some overall accuracy.
09 — Discussion & Limitations

What We Learned — and What We Didn't

Key Findings

  • Classical machine learning can detect sarcasm with meaningful accuracy (F1: 0.787, AUC: 0.883) when trained on multi-domain data.
  • Word content signals are stronger within a single domain but do not generalise. Stylistic signals are weaker but more universal.
  • Combining both feature types consistently outperforms either alone.
  • Standard sentiment tools (VADER, TextBlob) structurally fail at sarcasm — they misclassify 31–50% of sarcastic texts as positive depending on domain.
  • Source leakage is a real and underappreciated risk in single-source datasets. A model can appear excellent while learning the wrong thing entirely.
  • Word pairs add minimal value over single words — sarcasm requires broader contextual understanding than two-word patterns can provide.

Limitations and Threats to Validity

  • Residual leakage: Even the combined model may retain some source-specific signals. A fully leakage-free model would require balanced multi-source data from many different publications and platforms.
  • Domain imbalance: The combined dataset is 80% news headlines and 20% forum comments. The model still sees more news-style sarcasm during training, which may bias it toward that style.
  • Text length mismatch: News headlines (~10 words) and forum comments (~35 words) differ structurally. StandardScaler mitigates length-based feature differences but does not eliminate them.
  • Classical model ceiling: The maximum achievable F1 with bag-of-words approaches is limited. Sarcasm ultimately requires understanding intent, tone, and cultural context — capabilities that transformer-based language models (e.g. BERT, GPT) would handle substantially better.
  • Binary classification: We treat all sarcasm as a single category. In reality, sarcasm ranges from mild irony to heavy satire, and different subtypes may require different detection approaches.
